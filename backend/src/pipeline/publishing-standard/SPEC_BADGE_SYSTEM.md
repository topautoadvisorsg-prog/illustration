# SPEC — Badge System Standardization (STD-2)

**Status:** draft — awaiting operator sign-off. No code until approved.
**Owner module:** Badge System (Standard v1.2). Owns badge taxonomy, visual
design, assets, and the stamping *contract*. Does NOT own colour (PALETTE),
geometry (Layout), or the stamping *act* (Print-Prep, STD-3).
**Rule Zero:** every badge colour is a PALETTE token reference, never a literal.

---

## 1. The problem this closes

v1.1 locked the badge *taxonomy* (8 region / 9 hazard / 5 source) and v1.2's
prompt tells the model to **leave the bottom corners clean**. But there is **no
visual** — no badge assets, no renderer. Today a generated page has reserved
empty corners and nothing to put in them. STD-2 produces the **stampable badge
visual** and the **contract** Print-Prep (STD-3) will consume.

---

## 2. Asset strategy — deterministic SVG renderer (NOT hand-drawn files)

**Decision:** badges are generated by a pure function `renderBadgeSvg(family,
value)`, composed from shape primitives + PALETTE tokens — **not** 21 hand-
authored PNG/SVG files.

Why:
- **Rule Zero / single source of colour** — the SVG pulls `PALETTE.*` hex; no
  file can drift from the Standard.
- **No 21-file drift** — one renderer, consistent geometry across every badge.
- **Crisp at any DPI** — vector → Print-Prep rasterizes at 300 DPI (or 600 for
  safety) at stamp time.
- **Testable** — assert the SVG contains the right token + icon, deterministically.

Lives in `pipeline/publishing-standard/badges/render-badge.ts`. Pure, no I/O.

---

## 3. Visual language — one shape per family (so they never read as each other)

| Family | Shape | Treatment | Colour source |
|---|---|---|---|
| **Region** | circular ring + centered icon + small-caps label below | engraved line ring, the v1.1 look | `PALETTE` region token (ring) + `ink` (icon/label) |
| **Hazard** | rounded **shield** + icon (distinct from the region circle) | heavier stroke, signals "alert" without screen-red | `PALETTE` hazard token (shield) + `ink` |
| **Source** | small **wax-seal** roundel, monochrome | subtle, smallest of the three | `PALETTE.ink` only |

All three share: warm-sepia engraving feel, no gradients, no digital shadows —
consistent with the Standard's ornament language.

---

## 4. Icon set (the one real design decision)

Each badge value needs a glyph. v1 proposes a **minimal engraved line-icon set**,
authored once as SVG path primitives, one per value:

- **Region:** FOREST=evergreen, MOUNTAIN=peaks, RIVER=wave, WETLAND=reed,
  COASTAL=wave+shore, ALPINE=bare summit, FIELD=grass tufts, GENERAL=compass rose.
- **Hazard:** DEADLY=skull, TOXIC=trefoil/leaf-x, VENOMOUS=fang, AGGRESSIVE=horns,
  CAUTION=exclamation, EXPERT_REVIEW=magnifier, EDIBLE=leaf-check, MEDICINAL=mortar+pestle.
- **Source:** small lettermark in the seal (S/F/T/H/G) or a tiny book/quill glyph.

Icons are simple, legible at ~0.4in, and drawn in `ink`. **Open question Q1:**
ship the minimal line-icon set as proposed, or commission richer engraved icons
first? (Recommend: ship minimal now, upgrade post physical-proof.)

---

## 5. In-corner composition (within Layout's reserved 0.9in safe zones)

Per `BADGE_PLACEMENT` (already locked): region → bottom-left, hazard →
bottom-right, source → bottom-right inner.

- **Bottom-left:** the single region badge.
- **Bottom-right:** the hazard **stack** — most-severe first (top/outermost),
  rendered in `HAZARD_DISPLAY_ORDER`, max **2** hazard badges shown (a 3rd is
  rare and would crowd the corner; overflow drops the least-severe). The source
  seal sits just inboard of / beneath the hazard.
- **NONE hazard** → no hazard badge stamped (corner stays clean).
- Badges never overlap the trim-safe area; Layout owns the exact rect, this
  SPEC owns the within-corner arrangement.

---

## 6. The stamping contract (what Print-Prep / STD-3 consumes)

Badge System exposes, per page:

```ts
interface StampableBadge {
  family: 'region' | 'hazard' | 'source';
  value: string;          // e.g. 'FOREST', 'DEADLY'
  svg: string;            // renderBadgeSvg(family, value)
  corner: 'bottom-left' | 'bottom-right';
  order: number;          // stack order within the corner (0 = outermost)
}
function badgesForPage(badgeSet: Badge[]): StampableBadge[];
```

Print-Prep's job (STD-3) is purely: rasterize each `svg` at target DPI and
composite at the Layout-computed rect for its `corner` + `order`. **No badge
logic leaks into Print-Prep** — it stamps what the Badge System hands it.

---

## 7. Sizing & DPI

- Badge nominal size: ~0.55in region circle, ~0.5in hazard shield, ~0.32in
  source seal (tunable constants in the Badge System).
- SVG is unit-agnostic; Print-Prep rasterizes at 300 DPI (configurable to 600).
- All sizes are Badge-System constants referencing the Standard's
  `BADGE_PLACEMENT.safeZoneIn` so they always fit the reserved corner.

---

## 8. Legend dependency (BDG-1)

A badge is meaningless without a key. The **front-matter SOURCES/legend page**
(Front Matter SPEC) renders the same `renderBadgeSvg` assets next to their
meaning + the "sourced, not field-tested" disclaimer. STD-2 makes the renderer;
Front Matter consumes it. Noted, not built here.

---

## 9. v1 scope vs deferred

**In v1:** `renderBadgeSvg` (all families/values, Palette-tokened), the icon
set, `badgesForPage` (the contract), sizing constants, tests.
**Deferred:** richer commissioned icons (post-proof), per-publisher badge styles
(multi-tenant Standard engine), animated/interactive badges (digital edition).

**Not built here:** the actual stamping (Print-Prep, STD-3) — STD-2 only
delivers the assets + contract it will call.

---

## 10. Tests

- `renderBadgeSvg` for every value returns valid SVG containing the correct
  PALETTE token hex and the right icon id; **no literal hex** in the source.
- Rule Zero: grep the renderer source — zero hardcoded `#hex` (all via PALETTE).
- `badgesForPage`: ordering (severe-first), NONE→no hazard badge, max-2 hazard
  cap, region always present, source always present.
- Corner assignment matches `BADGE_PLACEMENT`.
